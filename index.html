如何实现一套iOS-router

发表于 2017-04-21 | 分类于 IOS |

router是什么？

简单的说，router就是用url跟一些操作绑定，我们可以通过我们设定的url，执行设定的操作，如打开某个页面。看图：

为什么要router？

随着我们的项目越来越大，界面越来越多，逻辑越来越复杂。界面间的路由也变得越来越臃肿，耦合越来越大。我们需要一套框架来解决和简化这些问题。也就是说我们需要router。router主要作用：

  1. 解耦界面间的越来越复杂的依赖关系。
  2. 统一接口.
  3. 实现router拦截器。集中处理需要拦截的需求，如登录拦截。也方便埋点。
  4. 方便和其他app的通信，方便处理web跳转到app的逻辑。

如何实现router

  1. 明确我们需要做什么。

    • 我们需要一个register来注册我们的类，如ViewController。绑定url和类。
    • 我们需要一个router来处理我们所有的路由逻辑。因此要定义好router的接口。
    • 我们需要拦截器。因此要定义好拦截器接口。
    • 我们需要至少支持push和present两种模式。

  2. 实现方案

  • 在register模块上，保存url和类的映射关系。并保存所有的拦截器。

  • 在router模块

    • 由于有两种模式，因此得有两个方法调用present和push（个人感觉navigate更合适，因为可以往前或往后导航）
    • 由于可以前后导航，因此得自定义的vc类中，得定义两个接口，init和update。
    • 维护路由堆栈，加入新的用init，导航到堆栈中存在的就调update。
  • 在跳转之前，调用所有的拦截器，根据拦截器返回的结果再做真正的跳转操作。
  • 用的是字典传参。
  • 由于用的是字典传参，最好加入依赖注入方案。

先假想使用router，再思考封装router。

在上面的方案中，我们已经定义好了我们的router中的一些内容，大概构思了一个方案。我们先想想如果我们已经有了一套router，我们该怎么使用才满足我们的需求还有体验。（我的想法）

  1. 首先我们自定义一个vc，实现init和update协议。有依赖注入肯定最好，直接调用注入的方法，不用逐一赋值。
  2. 在register模块上注册我们的vc。有拦截器加拦截器。
  3. 最后调用router相应的方法传参和跳转到我们定义的vc吧。为了更好的使用体验(参数的不确定)，用builder模式好一点。

总结

以上是我对router的一些想法，也按照这个想法实现了一套router。再也不用关注路由的关系了，只需关注本身页面的构造。达到解耦的效果。特别在嵌套很深的界面中，路由可直接在操作中进行，再也不用回调到外面的controller进行跳转。总的来说，效果不错。

IOS 基础

发表于 2017-01-20 | 分类于 IOS |

1. 语法基础

1.1 面向对象

  • 类和对象

    .h文件

    .m文件

  • 属性，setter，getter

    OC2.0开始用@property

    @property标注的属性自动生成setter和getter方法

    .会调用getter和setter

    指示符：

    • assign：一般的赋值，基本类型，long，id等
    • atomic（nonatomic）：原子操作，多线程的同步，性能低，单线程用nonatomic，默认是atomic。

    • copy：setter的时候是赋值原来对象的副本。改变原来对象不影响setter对应的属性。

    • retain：在大多数情况下跟strong一样，在ARC下尽量不要用retain，用strong较好。

    • strong：保持引用，引用计数不为0不会被回收。
    • weak：可能会被回收。
    • readonly：只读，只生成getter。
    • readwrite：读写。

  • block

    是OC对C的扩展，在OC中应用广泛，虽然大多数情况下我们不用写block，但是很多OC的API和第三方框架的API都大量应用block。

    block作为参数时，只能是最后一个或者多个参数。

    循环引用问题

    __block在MRC中不会引起循环引用，但是在ARC中会引起循环引用。因此要用weak。

  • 类别（category）和扩展 （extension）

    类别是在一些已有的类中添加一些属性和方法。没必要用继承。多用于工具类。如：NSNumber。也可以把类的私有方法写在上面以调用私有方法。

    extension里面的方法是必须实现的。属性一般放在里面默认私有。可以说是匿名的类别。

  • 协议@protocol和委托delegate

    协议类似于java中的接口。用过协议实现delegate。

  • 单例

  • id和instancetype的区别
    • alloc和init本身会关联类型。
    • instancetype会让不关联类型的方法关联类型。

  • 内存管理

    • 现在用的是ARC机制管理内存，不用手动释放。ARC会自动在需要的地方添加retain和release代码。对象的引用计数为0的时候会被释放。
    • 在ARC下不能手动retain和release。
    • 使用block和delegate回调的时候注意循环引用问题。
    • 用底层Core Foundation对象的时候，仍然要用MRC的内存管理方式管理，但是目前来说很少用到。
    • 用Instruments检测。

  • NSNumber

  • NSString

  • NSArray

2. UI基础

  • UILabel(常用) 对应Android的TextView

  • UIButton（常用）对应与Android的Button+ImageButton

  • UITextField单行（常用）对应android的EditText

  • UITextView多行 对应android的EditText
  • UISwitch 对应于Android的switch
  • UISegmentControl Android没有，但是有类似的控件是RadioGrop
  • UIImageView（常用）对应Android的ImageView
  • UIProgressView 对应android的progressBar + seekBar
  • UIActivityIndicatorView（常用）对应android的自定义progressBar

  • UIAlertView和UIActionSheet（常用）(IOS9以后都被弃用)
    现在用的是UIAlertViewController（IOS8开始有的）。对应于android的dialog。

  • UIScrollView 对应android的scrollView

  • UIDatePicker日期选择 对应android的DatePicker
  • UIPickerView选择器 android的单选和多选框，但是样式完全不一样。
  • UIWebView（常用）对应android的webView
  • UIToolBar（常用）对应android的actionBar
  • UITableView（常用）和UITableViewController 对应android的listView
  • UISearchBar（常用）对应android的searchVIew
  • UISearchDisplayController
  • UINavigationBar和UINavigationBarController（常用）IOS特有，用于页面导航
  • UICollectionView和UICollectionViewController（常用） IOS特有，表格控件。
  • UITabBar和UITabBarController（常用）对应android的tabLayout
  • UIPageControl和UIPageViewController IOS特有，页控件

  • 生命周期（应用生命周期，Controller生命周期）

    这里讲Controller的生命周期。

    • initWithxxx 只执行一次
    • loadView：加载view 只执行一次，执行完这个才有view。
    • viewDidLoad：view加载完毕 只执行一次
    • viewWillAppear：控制器的view将要显示
    • viewWillLayoutSubviews：控制器的view将要布局子控件
    • viewDidLayoutSubviews：控制器的view布局子控件完成。
    • viewDidAppear:控制器的view完全显示
    • viewWillDisappear：控制器的view即将消失的时候
    • viewDidDisappear：控制器的view完全消失的时候
